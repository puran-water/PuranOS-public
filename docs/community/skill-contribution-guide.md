# The Skill Model

## Why this exists

Industrial engineering firms lose institutional knowledge every time a senior engineer leaves, changes roles, or simply forgets how they solved a problem three projects ago. Sizing procedures, equipment selection logic, deliverable formatting — all of it lives in people's heads rather than in systems.

Skills are PuranOS's mechanism for making that knowledge durable and machine-executable.

## What a skill is

A skill is a markdown file that encodes an operating procedure. It specifies which MCP tools to call, in what order, with what parameters, what validation checks to perform between steps, and what the output must look like.

A skill is not a prompt template. Prompt templates describe what you want. Skills describe how to get it reliably. The difference matters because repeatability and auditability are non-negotiable in engineering work.

A skill is not a script. Scripts are brittle — they break when inputs vary. Skills are declarative procedures that an agent executes within persona boundaries, adapting to the specific project context while following the defined workflow.

## Skill anatomy

Every skill has two parts: YAML frontmatter for metadata, and a structured markdown document for the workflow.

### Frontmatter

```yaml
name: mle-bioreactor-sizing
description: >
  Size an MLE bioreactor from influent characterization and effluent targets.
  Use when pe-lead delegates biological treatment sizing or when the user
  requests activated sludge design.
user-invocable: true
```

| Field | Purpose |
|---|---|
| `name` | Kebab-case identifier matching the skill directory name |
| `description` | When to activate — includes trigger phrases so the agent recognizes relevant requests |
| `user-invocable` | Whether a user can trigger the skill directly, or whether it is only called by other workflows |

### MCP Tool Map

Lists which tools are called, from which MCP servers, for what purpose. This scopes the skill to the tools available to its target persona — a skill cannot reference tools outside the persona's MCP configuration.

### Execution Workflow

Ordered steps with explicit tool calls. Each step describes what to do and what to verify before proceeding. Steps may include conditional branches (e.g., "if influent BOD/COD ratio is below 0.4, flag for supplemental carbon evaluation").

### Output Contract

Defines the required structure of the skill's output: mandatory sections, data provenance requirements (which tool produced each value), format specifications, and conditional sections included only when relevant.

### Risk Boundaries

What the skill must not do, when to escalate to a human reviewer, and what credibility constraints apply to inputs. A sizing skill operating on "preliminary" model inputs should flag that its outputs inherit that credibility level.

## Illustrative example: equipment list generation

An equipment list skill encodes the workflow a senior engineer follows when assembling an equipment list from sizing results:

1. **Query the design state.** Read the plant-state registry for the current process unit topology and stream conditions.

2. **Extract equipment from sizing results.** For each process unit, collect the equipment items generated by upstream sizing skills — pumps, blowers, tanks, membranes, heat exchangers.

3. **Validate each item.** Verify conformance to the equipment-item schema: correct tag format (ISA 5.1 area-code-sequence pattern), required attributes present, capacity units consistent with the process unit type.

4. **Assign tags.** Apply the tagging convention — area code from the ISA-95 hierarchy position, equipment code from the process-unit taxonomy, sequence number in order of appearance.

5. **Aggregate and cross-check.** Total connected load, total flow capacity, material consistency across the plant.

6. **Generate the deliverable.** Produce the equipment list with a typed artifact envelope declaring version, dependencies on upstream sizing artifacts, and conformance claims.

7. **Register the artifact.** Write the equipment list to the artifact registry so downstream artifacts (cost estimates, procurement packages, I&C schedules) can declare dependencies on it.

The skill does not contain the engineering knowledge to size the equipment — that lives in the simulation engines and their sizing skills. This skill contains the procedural knowledge to assemble, validate, tag, format, and register the results. That procedural knowledge is what disappears when a senior engineer moves on.

## How skills relate to the architecture

Skills sit at the intersection of several architectural layers:

| Relationship | Mechanism |
|---|---|
| Skills scope to personas | An equipment list skill executes under pe-deliverables, not pe-process. The persona's MCP configuration determines which tools are available. |
| Skills call MCP tools | Every tool reference in a skill must resolve to a tool on the target persona's MCP server list. |
| Skills produce schema-validated outputs | Equipment items conform to the equipment-item schema. Deliverables carry artifact envelopes. Credibility metadata propagates from inputs to outputs. |
| Skills compose | A cost estimation skill depends on the equipment list skill's output. The artifact registry tracks these dependencies explicitly. |
| Skills evolve | Each project execution may refine a skill — better heuristics, new edge cases, improved validation checks. The skill library improves over time. |

## Why domain expertise is the bottleneck

The hardest part of building a skill is not the markdown formatting or the tool call syntax. It is knowing the right workflow: which parameters to check first, what heuristics apply to a given equipment type, when conservative assumptions are warranted, what a reviewer expects to see in the output.

That knowledge comes from years of sizing treatment systems, managing procurements, or producing engineering deliverables. It cannot be extracted from documentation or inferred from code. It is the accumulated judgment of practitioners who have done the work repeatedly and learned what produces reliable results.
